Core Fundamentals
SQL, data modeling, and architectural concepts
🗃️ SQL Deep Dive Critical
+

SQL is the backbone of data engineering. You must know it deeply — not just basic SELECT statements.

Window Functions — Operate over a partition of rows without collapsing them (unlike GROUP BY):

-- Rank employees by salary within each department SELECT name, department, salary, RANK() OVER (PARTITION BY department ORDER BY salary DESC) AS rnk, LAG(salary) OVER (PARTITION BY department ORDER BY salary) AS prev_salary, SUM(salary) OVER (PARTITION BY department) AS dept_total FROM employees;

CTEs (Common Table Expressions) — Make complex queries readable:

WITH daily_sales AS ( SELECT date, SUM(amount) AS total FROM orders GROUP BY date ), rolling_avg AS ( SELECT date, total, AVG(total) OVER (ORDER BY date ROWS BETWEEN 6 PRECEDING AND CURRENT ROW) AS avg_7d FROM daily_sales ) SELECT * FROM rolling_avg;

JOINs to remember: INNER (intersection), LEFT/RIGHT (keep one side), FULL OUTER (union), CROSS (cartesian), SELF JOIN, ANTI JOIN (NOT EXISTS / NOT IN).

Query Optimization: Use EXPLAIN ANALYZE, add indexes on JOIN/WHERE columns, avoid SELECT *, use partitioning, push filters early, avoid functions on indexed columns in WHERE clauses.

📐 Data Modeling Important
+

Normalization reduces redundancy. Key forms:

Normal FormRuleExample Issue Fixed
1NFAtomic values, no repeating groupsNo comma-separated lists in a column
2NFNo partial dependencies on composite PKRemove cols depending on only part of PK
3NFNo transitive dependenciesDept_Name depending on Dept_ID, not Employee

Star Schema (denormalized, fast reads): One central Fact table surrounded by Dimension tables. Preferred in data warehouses.

Snowflake Schema: Dimensions are further normalized into sub-dimensions. Saves storage but slower query performance due to more JOINs.

Slowly Changing Dimensions (SCD):

Type 0 – Never change Type 1 – Overwrite Type 2 – Add new row (history) Type 3 – Add new column Type 4 – History table
Interview Tip: SCD Type 2 is the most commonly asked. Know how to implement it: add effective_from, effective_to, and is_current columns.

🔍 Indexing Strategies Important
+

B-Tree Index: Default index type. Good for equality and range queries. Works on most data types.

Hash Index: Only for equality comparisons. Faster than B-tree for exact matches but useless for ranges.

Bitmap Index: Great for low-cardinality columns (gender, status). Used in data warehouses.

Columnar Index / Clustered Index: Data physically stored in index order. Only one per table. Huge performance boost for range scans.

Composite Index: Index on multiple columns. Order matters! Rule: put most selective column first. The leftmost prefix rule applies.

When NOT to index: Small tables, columns with high write frequency, columns rarely used in WHERE/JOIN, columns with very low selectivity.

✈️ Apache Airflow Critical
+

Airflow is the most popular pipeline orchestrator. It defines workflows as DAGs (Directed Acyclic Graphs) using Python.

Key Concepts:

DAG – workflow definition Task – unit of work Operator – task type Sensor – waits for condition XCom – task communication Scheduler – triggers runs Executor – runs tasks Hooks – connection abstractions
from airflow import DAG from airflow.operators.python import PythonOperator from datetime import datetime def extract(): pass # pull from source def transform(): pass # clean & reshape def load(): pass # write to warehouse with DAG('my_etl', start_date=datetime(2024,1,1), schedule_interval='@daily', catchup=False) as dag: t1 = PythonOperator(task_id='extract', python_callable=extract) t2 = PythonOperator(task_id='transform', python_callable=transform) t3 = PythonOperator(task_id='load', python_callable=load) t1 >> t2 >> t3 # define dependency chain

Idempotency: A critical principle — running a pipeline multiple times should produce the same result. Always design for re-runs.

📦 Data Ingestion Patterns Important
+

Full Load: Replace entire table every run. Simple but costly for large datasets.

Incremental Load: Only load new/changed records. Requires a watermark (updated_at, id). Much more efficient.

CDC (Change Data Capture): Capture DB changes (INSERT/UPDATE/DELETE) in real-time using database logs. Tools: Debezium, AWS DMS.

Micro-batch: Small batches every few minutes. Compromise between batch and streaming.

-- Incremental load pattern with watermark INSERT INTO warehouse.orders SELECT * FROM source.orders WHERE updated_at > (SELECT MAX(updated_at) FROM warehouse.orders)
🌊 Streaming with Kafka Advanced
+

Apache Kafka is a distributed event streaming platform. Think of it as a persistent, high-throughput message bus.

Core Components:

ComponentWhat It Does
ProducerPublishes messages/events to topics
ConsumerReads messages from topics
TopicCategory/stream name for messages
PartitionOrdered, immutable log — unit of parallelism
BrokerKafka server that stores & serves data
Consumer GroupMultiple consumers sharing partition reads
OffsetPosition of a message in a partition
Key insight: Kafka retains messages for a configurable period. Consumers control their own offset — they can replay events from the beginning. This makes it different from traditional queues.

Delivery Guarantees: At-most-once (may lose), At-least-once (may duplicate), Exactly-once (hardest, requires idempotent producers + transactions).

🔧 dbt (data build tool) Modern Must-Know
+

dbt transforms raw data in your warehouse using SQL SELECT statements. It brings software engineering practices (version control, testing, docs) to data transformation.

dbt Model types: source (raw tables) → staging (light cleaning) → intermediate (joins/reshaping) → marts (business-ready).

-- models/staging/stg_orders.sql {{ config(materialized='view') }} SELECT order_id, customer_id, CAST(created_at AS DATE) AS order_date, UPPER(status) AS order_status FROM {{ source('raw', 'orders') }}

Materializations: view (query every time), table (persisted), incremental (append/merge new rows only), ephemeral (CTE, no object created).

🔑 ACID vs BASE Concepts
+

ACID (Relational DBs — PostgreSQL, MySQL):

Atomicity – all or nothing Consistency – valid state always Isolation – concurrent txns separated Durability – committed = persisted

BASE (NoSQL — Cassandra, DynamoDB):

Basically Available Soft state Eventually Consistent
CAP Theorem: A distributed system can only guarantee 2 of 3: Consistency, Availability, Partition Tolerance. NoSQL systems typically choose AP (available + partition tolerant) over CP.
📊 Columnar vs Row Storage Key Concept
+

Row-oriented storage (PostgreSQL, MySQL): All columns for a row stored together. Fast for INSERT/UPDATE of individual records. Good for OLTP.

Column-oriented storage (Redshift, BigQuery, Parquet): All values of a column stored together. Excellent for analytical queries that touch few columns across many rows. Better compression (similar values together).

Parquet: The de facto standard columnar file format for big data. Self-describing schema, efficient compression, predicate pushdown. Use Parquet for your data lake files — always.

File formats comparison:

FormatTypeBest For
CSVRowSmall data, human readability
JSONRowSemi-structured, APIs

ORCColumnarHive, HDFS workloads
AvroRowSchema evolution, Kafka serialization
DeltaColumnar + ACIDLakehouse, UPSERT support
🗂️ Partitioning & Bucketing Performance
+

Partitioning: Divides data into separate physical folders/files based on a column value (usually date). Queries filtering on partition column skip irrelevant data entirely — this is called partition pruning.

-- Partition by date in Hive/Spark SQL CREATE TABLE events PARTITIONED BY (event_date STRING) STORED AS PARQUET; -- Only scans 2024-01-15 partition folder SELECT * FROM events WHERE event_date = '2024-01-15'

Bucketing (Clustering): Distributes data into a fixed number of buckets using a hash of a column. Useful for JOIN performance when both tables are bucketed on the same column.

Over-partitioning trap: Too many small partition files = "small file problem." A good target is partition files of 128MB–1GB.

⚡ Apache Spark Deep Dive Critical
+

Spark is the dominant distributed data processing engine. It processes data in-memory, making it 10-100x faster than Hadoop MapReduce.

Core Concepts:

ConceptDescription
RDDResilient Distributed Dataset — low-level, immutable, fault-tolerant distributed collection
DataFrameDistributed table with named columns + schema. The modern API (like pandas but distributed)
DatasetType-safe DataFrame (Scala/Java)
DriverOrchestrates the job, holds SparkContext
ExecutorWorker node that runs tasks
DAG SchedulerConverts job into stages and tasks

Transformations (lazy) vs Actions (triggers execution):

from pyspark.sql import SparkSession from pyspark.sql.functions import col, sum, avg spark = SparkSession.builder.appName("example").getOrCreate() df = spark.read.parquet("s3://bucket/data/") # Lazy transformations (nothing runs yet) result = (df .filter(col("status") == "active") .groupBy("region") .agg(sum("revenue").alias("total_rev")) ) result.show() # ACTION — triggers execution

Partitions: Spark splits data into partitions processed in parallel. Rule of thumb: 2-4 partitions per CPU core. Use repartition() or coalesce() to adjust.

Shuffling: The most expensive operation. Happens during groupBy, join, distinct. Minimize shuffles by caching, broadcast joins, and partition design.

🔗 Spark Joins & Optimization Advanced
+

Broadcast Join: For small + large table joins. Small table is broadcast to all nodes, eliminating shuffle. Use when one table is <~200MB.

from pyspark.sql.functions import broadcast # Hint to broadcast small lookup table result = large_df.join(broadcast(small_df), "customer_id")

Sort-Merge Join: Default for large tables. Both tables partitioned, sorted, then merged. Causes shuffle.

Skew Handling: When one partition has vastly more data than others (skew), that task becomes a bottleneck. Solutions: salting keys, skew hints, AQE (Adaptive Query Execution) in Spark 3.x.

Caching / Persistence:

# Cache: stores in memory (default) df.cache() # Persist with explicit storage level from pyspark import StorageLevel df.persist(StorageLevel.MEMORY_AND_DISK)

🔐 Data Governance & Security Important
+

PII (Personally Identifiable Information): Names, emails, SSNs. Must be protected. Techniques: encryption, masking, tokenization, pseudonymization.

Row-Level Security (RLS): Users can only see rows relevant to them (e.g., sales rep sees only their region's data).

Column-Level Security: Restrict access to sensitive columns like salary, SSN.

Data Catalog: Metadata management — what data exists, where, who owns it, what it means. Tools: Apache Atlas, AWS Glue Data Catalog, Alation, Collibra.

GDPR/CCPA Compliance: Right to erasure (delete user data), data minimization, audit trails. In a data lake, this is hard — use Delta Lake's DELETE support or maintain a deletion log.

FUNDAMENTALS
Q01What is the difference between ETL and ELT?▾
ETL (Extract-Transform-Load) transforms data before loading it into the target. ELT (Extract-Load-Transform) loads raw data first, then transforms it inside the warehouse. ELT is now preferred with powerful cloud warehouses (BigQuery, Snowflake) because the warehouse itself handles heavy computation efficiently. ETL is still used when data needs to be cleaned/masked before storage for security reasons.
Q02Explain SCD Type 2 and when to use it.▾
SCD Type 2 (Slowly Changing Dimension Type 2) tracks historical changes by adding a new row for each change rather than overwriting. You add columns: effective_from (start date), effective_to (end date or NULL for current), and is_current (boolean flag). Use it when you need to analyze data as it was at a point in time — e.g., "what was the customer's address when they placed this order?"
Q03What is idempotency in data pipelines?▾
An idempotent pipeline produces the same result regardless of how many times it runs. This is critical for reliability — if a pipeline fails and reruns, it shouldn't duplicate data or produce incorrect results. Implementation patterns: use UPSERT (merge) instead of INSERT, delete-then-insert for a date partition, or write to a staging table and swap atomically.
Q04What is a DAG in Apache Airflow?▾
A DAG (Directed Acyclic Graph) is the core Airflow object representing a workflow. "Directed" means tasks flow in one direction (no circular dependencies). "Acyclic" means no cycles — a task can't be its own ancestor. Each node is a Task (unit of work) and edges define dependencies. DAGs are defined in Python and scheduled via cron expressions or data intervals.
Q05Difference between Data Warehouse and Data Lake?▾
A Data Warehouse stores structured, processed data with a predefined schema (schema-on-write). It's optimized for SQL analytics and BI. Cost is high. A Data Lake stores raw data in any format (CSV, JSON, Parquet, images, logs) at low cost. Schema is applied at query time (schema-on-read). It's ideal for ML and data science. The modern Lakehouse architecture combines both — cheap lake storage with warehouse-quality query performance via formats like Delta Lake.
SQL
Q06Write a SQL query to find the second highest salary.▾
-- Method 1: DENSE_RANK (handles ties correctly) SELECT salary FROM ( SELECT salary, DENSE_RANK() OVER (ORDER BY salary DESC) AS rnk FROM employees ) t WHERE rnk = 2; -- Method 2: Subquery (simpler) SELECT MAX(salary) FROM employees WHERE salary < (SELECT MAX(salary) FROM employees);
DENSE_RANK is preferred because it handles the case where multiple employees share the highest salary.
Q07What is the difference between RANK, DENSE_RANK, and ROW_NUMBER?▾
For values [100, 100, 90, 80]: ROW_NUMBER gives unique sequential numbers: 1, 2, 3, 4 (arbitrary for ties). RANK gives: 1, 1, 3, 4 (gap after ties — next rank = position). DENSE_RANK gives: 1, 1, 2, 3 (no gap — consecutive ranks even after ties). Use DENSE_RANK when you want "nth highest" without gaps. Use ROW_NUMBER when you need unique row identifiers.
Q08What is a CTE and when would you use it?▾
A CTE (Common Table Expression) is a named temporary result set defined with WITH. It exists only for the duration of the query. Use cases: (1) Break complex queries into readable steps, (2) Recursive queries (hierarchies, org charts), (3) Reference the same subquery multiple times. Unlike subqueries, CTEs can be recursive and are generally easier to read and debug. dbt models are essentially named CTEs stored as views/tables.
Q09How do you detect and handle duplicate records in SQL?▾
-- Detect duplicates SELECT email, COUNT(*) FROM users GROUP BY email HAVING COUNT(*) > 1; -- Remove duplicates, keep latest WITH deduped AS ( SELECT *, ROW_NUMBER() OVER ( PARTITION BY email ORDER BY created_at DESC ) AS rn FROM users ) SELECT * FROM deduped WHERE rn = 1;
Q10Explain the difference between WHERE and HAVING.▾
WHERE filters individual rows before grouping. HAVING filters groups after GROUP BY aggregation. You can't use aggregate functions (SUM, COUNT) in WHERE — use HAVING for that. Example: WHERE salary > 50000 filters rows. HAVING COUNT(*) > 5 keeps only departments with more than 5 employees after grouping. Execution order: FROM → WHERE → GROUP BY → HAVING → SELECT → ORDER BY → LIMIT.
SPARK & BIG DATA
Q11What is a Spark shuffle and why is it expensive?▾
A shuffle happens when Spark needs to redistribute data across partitions — typically during groupBy, join, distinct, and repartition. It involves writing data to disk on each executor, transferring over the network, and reading it back. This is expensive because: (1) disk I/O, (2) network transfer, (3) sort/merge operations. To minimize shuffles: use broadcast joins for small tables, cache DataFrames that are reused, design partitioning strategy upfront, and enable Spark AQE (Adaptive Query Execution).
Q12What is data skew in Spark and how do you handle it?▾
Data skew occurs when data is unevenly distributed across partitions — one partition has vastly more data than others. That "hot" partition becomes the bottleneck, and the entire job waits for it. Solutions: (1) Salting — add a random prefix to the skewed key before join, then aggregate twice. (2) Spark 3.x AQE skew join optimization — automatically splits skewed partitions. (3) Broadcast join — if one table is small enough. (4) Pre-aggregate before join to reduce data size. Identifying skew: check Spark UI for long-running stages.
Q13What is the difference between repartition() and coalesce() in Spark?▾
Both change the number of partitions. repartition(n) creates n partitions with a full shuffle — data is evenly redistributed. Use to increase partitions or redistribute skewed data. coalesce(n) reduces partitions by merging them on the same node — no full shuffle, but can result in uneven partitions. Use coalesce when reducing partitions (e.g., before writing to avoid too many small files). Rule: coalesce to reduce, repartition to increase or rebalance.
Q14Explain Kafka's consumer group concept.▾
A consumer group is a set of consumers that collectively consume a topic. Kafka assigns each partition to exactly one consumer within a group — enabling parallel processing. If a topic has 4 partitions and a group has 2 consumers, each consumer gets 2 partitions. If you add a 3rd consumer, one consumer gets 2 partitions and two get 1. If you add a 5th consumer, one is idle (can't have more consumers than partitions in a group). Multiple independent consumer groups can each read the entire topic independently (e.g., one group for analytics, another for notifications).
SYSTEM DESIGN
Q15Design a real-time analytics pipeline for 1M events/second.▾
Architecture: Producers → Kafka (partitioned heavily, e.g., 100 partitions) → Flink/Spark Streaming consumers → write to two sinks: (1) Redis/Druid for real-time dashboards, (2) S3/GCS (via exactly-once writes) for historical storage. Key considerations: partition Kafka by a balanced key (e.g., user_id hash), use compression (Snappy), set appropriate retention. For the streaming job: use watermarks to handle late data, use tumbling/sliding windows for aggregations. Write to columnar format (Parquet) in micro-batches for the historical sink.
Q16How would you handle late-arriving data in a streaming pipeline?▾
Use watermarks — a mechanism to declare how late data is expected to arrive. A watermark of 10 minutes means: for an event with timestamp T, hold the window open until the system's processing time reaches T + 10 min. Data arriving after the watermark is dropped or sent to a side output for reprocessing. In Spark Structured Streaming: df.withWatermark("event_time", "10 minutes"). For critical pipelines, combine with a late data reconciliation job that reprocesses a lookback window on a schedule (e.g., re-aggregate the last 24 hours every hour).
Q17What is the medallion architecture?▾
The medallion architecture (popularized by Databricks) organizes a data lake into three layers: Bronze (raw data exactly as ingested — no transformations, full history), Silver (cleaned, deduplicated, validated data — close to 3NF), and Gold (business-level aggregates and models ready for BI tools and ML). Each layer builds on the previous, and you can always reprocess from Bronze. It provides clear data lineage, quality guarantees at each layer, and separation of concerns.
Q18How do you optimize a slow running SQL query?▾
Step 1: Use EXPLAIN/EXPLAIN ANALYZE to see the query plan — look for Sequential Scans, high row estimates, nested loops. Step 2: Add indexes on JOIN keys and WHERE filter columns. Step 3: Avoid SELECT * — fetch only needed columns (enables columnar pushdown). Step 4: Push WHERE filters as early as possible in CTEs. Step 5: Avoid functions on indexed columns in WHERE (e.g., WHERE LOWER(email) — prevents index use). Step 6: Consider partitioning for large tables filtered by date. Step 7: Update table statistics (ANALYZE). Step 8: Check for missing JOINs causing cartesian products.
Q19What is CDC and how does Debezium work?▾
CDC (Change Data Capture) captures row-level changes (INSERT/UPDATE/DELETE) in a database as they happen. Debezium is an open-source CDC tool that reads database binary logs (MySQL binlog, PostgreSQL WAL, MongoDB oplog). It publishes changes as structured events to Kafka topics. Consumers downstream can then react to individual row changes in near-real-time. This is far more efficient than polling ("give me all rows where updated_at > X") for high-frequency change capture. Common use cases: database replication, cache invalidation, audit logs, event-driven microservices.
Q20What is the difference between a star and snowflake schema?▾
In a star schema, dimension tables are denormalized — all attributes for a dimension are in one table. In a snowflake schema, dimension tables are further normalized — e.g., a Product dimension might reference a Category table, which references a Department table. Star schema: fewer JOINs, faster queries, more storage. Snowflake schema: more JOINs, slower queries, less storage redundancy, easier maintenance. In practice, most data warehouses use star schemas for performance. The query slowdown of snowflake rarely justifies the storage savings with modern cheap cloud storage.

3. Mention trade-offs — every design decision has pros and cons. Showing this maturity stands out.
4. Know your resume — every technology you listed should be explainable at 3 levels of depth.
5. Idempotency & data quality come up in almost every pipeline design question — always address them.
6. For SQL: write the query, then think about performance (indexes, explain plan).
7. Brush up on EXPLAIN ANALYZE in PostgreSQL and .explain() in Spark — shows you think like a practitioner.
